## 关于本次测试

本测试围绕JD中权重最高的几项能力设计：Linux/中间件故障排查、自动化脚本、监控与On-Call响应思维。云平台实操（阿里云/火山云）、CI/CD（Jenkins）、K8s管理等经验会在后续面试沟通中了解，不在此次测试范围内，不必为此担心。

**关于时间**：下面每个任务标注的时间是我们对完成大致需要多久的估计，合计约3.5小时（任务一~三为必做，任务四选做，可加分）。这不是一个精确计时的考核项，也不会拿来打分——标注这个数字，只是想让你清楚这份测试大概是什么量级，不希望你因为想"做到完美"而额外投入很多时间（比如占用一整个周末）。多花二三十分钟把某部分想清楚完全没问题；但如果整体已经明显超出这个量级还没有进展，建议先停下来，把卡在哪里、你的思路写清楚提交即可，这也是我们想看到的真实状态。可以分多次完成，不要求一次性连续做完。

**关于工具使用**：欢迎使用任何你平时会用的工具，包括AI编程助手（Claude / Cursor / Copilot 等）、搜索引擎、文档。我们关心的是判断力和排查逻辑，而不是"是否独立完成"。请在报告中说明哪些地方用了AI辅助、AI的建议是否正确、你做了哪些验证或修改——这本身也是我们想考察的能力之一。

## 任务一：故障排查与修复（预估约90分钟，供参考）

**场景**：你拿到一个内部小服务"日报服务"（daily-report-service）的代码仓库，基于 Flask + MySQL + Redis，前面挂了一层 Nginx 反向代理。产品同事反馈"服务时好时坏，有时候打不开，有时候很慢"。

**要做的事**：

1. `docker-compose up` 启动整套环境。
2. 访问 `/health` 及几个业务接口，定位环境中存在的问题（不止一个）。
3. 逐一排查、定位根因、修复，直到服务稳定运行。
4. 提交：
   - 修改后的配置/代码文件
   - 一份排查报告（Markdown/txt均可），需包含：
     - 观察到的现象（日志片段或截图）
     - 排查思路：做了什么假设、如何验证、走过什么弯路（弯路不扣分，写出来反而更好）
     - 每个问题的根因
     - 修复方式，以及为什么选这种修法而不是别的方式
     - 若是生产环境，你会如何预防同类问题再次发生（1-2句话即可）

> 我们更看重排查过程的逻辑性，而不只是"最后修好了"。

## 任务二：自动化运维脚本（预估约60分钟，供参考）

用 Shell 或 Python 编写脚本，实现下面三项中的**至少两项**（三项都做有加分）：

**A. 日志巡检脚本** — 统计指定日志目录中最近 N 分钟内 ERROR/WARN 出现次数，超过阈值输出告警（打印或写文件即可，不需要真的接入IM/邮件）。阈值、时间窗口、目录需可配置，不要硬编码。

**B. 服务健康检查脚本** — 对一组服务地址逐个做健康检查，失败自动重试（次数可配置），最后输出一份健康报告（正常/异常/异常原因）。

**C. 磁盘清理脚本** — 检查指定目录的磁盘占用，超过阈值时按最早修改时间清理旧文件直到降到阈值以下；需要支持"演练模式"（只打印会删除什么，不真删）和"执行模式"。

**要求**：
- 有基本的异常处理，遇到边界输入不能直接崩溃
- 关键逻辑写注释说明为什么这么做（尤其AI辅助生成的部分，说明你验证/修改了什么）
- 简单说明你是怎么测试这个脚本的

## 任务三：监控告警方案设计（预估约40分钟，供参考，纯书面，无需搭环境）

假设任务一中的"日报服务"要上线到生产环境：

1. 若用 Prometheus + Grafana 做监控，你会重点关注哪些指标？列出**至少8个**具体指标（覆盖主机/应用/中间件三层），并说明为什么选它们。
2. 你会怎么给告警分级（哪些要立刻On-Call响应，哪些工作时间内处理即可）？举 2-3 个具体阈值例子。
3. 凌晨3点收到一条"数据库连接数接近上限"的告警，描述你从收到告警到解决（或上报）的完整处理步骤。
